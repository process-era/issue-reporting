## What makes a good bug report?

A useful bug report should include:

1. A clear summary
2. Steps to reproduce
3. Expected behavior
4. Actual behavior
5. Environment information
6. Screenshots, videos, logs, or console output if helpful

Please describe the issue in a way that someone without your local setup can understand and reproduce.

---

## Security and sensitive data

Do **not** report security vulnerabilities publicly.

If your report contains sensitive information, credentials, customer data, private process data, or security-related details, please contact the maintainers directly instead of opening a public issue.

Never include:

- Passwords
- API keys
- Access tokens
- Private customer names or process data
- Internal URLs that should not be public
- Database dumps
- Production logs with personal data
